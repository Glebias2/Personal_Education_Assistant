# RAG-система AI-ассистента курса — Отчёт

## 1. Общее описание

RAG (Retrieval-Augmented Generation) — подход, при котором LLM отвечает на вопросы пользователя, опираясь на релевантные фрагменты из загруженных материалов курса. Это позволяет AI-ассистенту давать точные ответы по конкретному учебному материалу, а не полагаться только на общие знания модели.

### Архитектура

```
Студент → Вопрос → Hybrid Search (Weaviate) → Топ-N чанков → LLM (Gemini) → Ответ
                         ↑                                          ↑
                   Векторное хранилище                    Системный промпт
                   (материалы курса)                     (роль преподавателя)
```

### Стек технологий

| Компонент | Технология |
|---|---|
| Векторная БД | Weaviate 1.29 + text2vec-transformers (multilingual) |
| Векторизация | Серверная (Weaviate, модель sentence-transformers) |
| LLM | Google Gemini (через LangChain) |
| Оркестрация | LangChain (ChatPromptTemplate, цепочки) |
| SQL БД | PostgreSQL (метаданные файлов, история чатов) |
| Фронтенд | React + react-markdown (рендеринг Markdown-ответов) |

## 2. Пайплайн: от загрузки документа до ответа

### 2.1. Индексация документов

**Файл:** `server/src/rag/indexer.py`

1. Преподаватель загружает файл (PDF, DOCX, TXT) через UI
2. Файл читается соответствующим лоадером (PyPDFLoader, Docx2txtLoader, TextLoader)
3. Текст разбивается на чанки через `RecursiveCharacterTextSplitter`:
   - **chunk_size = 1000** символов (полные абзацы с определениями)
   - **chunk_overlap = 200** символов (контекст не теряется на границах)
   - Разделители: `\n\n` → `\n` → `. ` → ` ` (сначала по абзацам, потом по предложениям)
4. Каждому чанку присваивается `file_id` (UUID) для привязки к файлу
5. Чанки сохраняются в Weaviate-коллекцию `VectorStorage_{storage_id}`
6. Weaviate автоматически векторизирует текст через `text2vec-transformers`
7. Метаданные файла сохраняются в PostgreSQL (`course_files`)

### 2.2. Поиск релевантных фрагментов

**Файл:** `server/src/database/vector/repositories/storage.py`

Используется **гибридный поиск** (Hybrid Search), объединяющий два подхода:

| Подход | Как работает | Для чего |
|---|---|---|
| **BM25** (ключевые слова) | Точное совпадение терминов | Находит конкретные термины, формулы, названия |
| **Vector Search** (семантика) | Cosine similarity эмбеддингов | Понимает смысл вопроса, даже если слова другие |

**Параметры гибридного поиска:**
- `alpha = 0.75` — 75% семантика, 25% ключевые слова
- `fusion_type = RELATIVE_SCORE` — нормализация скоров обоих методов
- `k = 15` — извлекаем 15 кандидатов
- `target_vector = "main_vector"` — именованный вектор в Weaviate

**Пример:** На вопрос "Что такое ОМТ?" BM25 находит чанки с точным вхождением "ОМТ", а vector search находит чанки где описывается "Object Modeling Technique" без аббревиатуры. Гибридный поиск объединяет оба результата.

### 2.3. Выбор контекста для LLM

**Файл:** `server/src/rag/chain.py`

Из 15 кандидатов берутся **топ-8 чанков** (уже отсортированы по релевантности Weaviate). Проверяется что суммарный текст содержит более 50 символов (защита от пустых чанков).

**Решение RAG vs General:**
- Если есть содержательные чанки → RAG chain (ответ на основе материалов)
- Если чанков нет или они пустые → General chain (свободный разговор)

### 2.4. Генерация ответа

**RAG Chain** — LLM получает:
1. Системный промпт (роль преподавателя + правила ответа)
2. Контекст из материалов (до 8 чанков, разделённых `---`)
3. Историю диалога (последние 20 сообщений)
4. Вопрос студента

**General Chain** — LLM получает:
1. Системный промпт (роль дружелюбного ассистента)
2. Историю диалога
3. Вопрос студента

## 3. Ключевые решения и почему они приняты

### 3.1. Persona-based промпт вместо "ответь по контексту"

**Было:**
```
Отвечай СТРОГО на основании предоставленного контекста из материалов курса.
```

**Проблема:** LLM буквально следовала инструкции — ссылалась на "контекст", "документ", писала "в контексте нет данных".

**Стало:**
```
Ты — опытный университетский преподаватель.
НЕ ссылайся на «контекст», «документ», «предоставленные материалы» — просто объясняй.
```

**Результат:** LLM отвечает как знающий преподаватель, не выдавая что читает из документа.

### 3.2. Гибридный поиск вместо чистого vector search

**Было:** Только `near_text` (семантический поиск).

**Проблема:** Multilingual модели дают высокие distance даже для хороших совпадений. Точные термины (аббревиатуры, формулы) могли не находиться.

**Стало:** `hybrid()` с `alpha=0.75` — BM25 ловит точные термины, vector search ловит смысл.

### 3.3. Убран жёсткий порог distance

**Было:** `DISTANCE_THRESHOLD = 0.45/0.65` — чанки с distance выше порога отбрасывались.

**Проблема:** Для multilingual моделей distance 0.3-0.8 — нормальный диапазон. Порог отбрасывал релевантные чанки.

**Стало:** Берутся топ-8 чанков по score гибридного поиска без жёсткого отсечения. LLM сама определяет что релевантно.

### 3.4. Увеличенные чанки (1000 символов)

**Было:** `chunk_size = 200` — 1-2 предложения. Определения и абзацы обрезались.

**Стало:** `chunk_size = 1000`, `chunk_overlap = 200` — полные абзацы с контекстом.

### 3.5. Markdown-форматирование ответов

LLM инструктирована использовать Markdown (заголовки, списки, жирный текст). На фронтенде ответы ассистента рендерятся через `react-markdown` с Tailwind Typography (`prose`).

## 6. Хранение данных

### PostgreSQL

- **chats** — id, student_id, course_id, name, created_at
- **chat_messages** — id, chat_id, role (human/ai), content, created_at
- **course_files** — id, course_id, filename, file_id, created_at
- **courses.storage_id** — ссылка на Weaviate-коллекцию

### Weaviate

- Коллекция `VectorStorage_{storage_id}` для каждого курса
- Свойства объекта: `text` (содержимое чанка), `file_id` (привязка к файлу)
- Вектор: `main_vector` (text2vec-transformers, серверная векторизация)

## 7. Возможные дальнейшие улучшения

| Улучшение | Описание | Сложность |
|---|---|---|
| **Parent-Child chunking** | Индексировать мелкие чанки (200 токенов) для точного поиска, возвращать большие (1000 токенов) для контекста | Средняя |
| **Contextual chunking** | При индексации добавлять к каждому чанку контекст ("Из главы 3, раздел ОМТ") через LLM | Средняя |
| **Cross-encoder reranking** | После retrieval прогонять через cross-encoder (bge-reranker-v2-m3) для точной переранжировки | Средняя |
| **HyDE** | Генерировать гипотетический ответ и искать по его эмбеддингу — лучше для сложных вопросов | Низкая |
| **Multi-query retrieval** | Генерировать 3-5 переформулировок вопроса и объединять результаты | Низкая |
| **Обновление модели эмбеддингов** | Заменить sentence-transformers на BGE-M3 или multilingual-e5-large (лучше для русского) | Высокая |
